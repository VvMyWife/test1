核心技术主线
1️⃣ OCR与结构化解析（MinerU方向）
研究 MinerU 如何完成 PDF/扫描件的版面解析（layout）+ 文本识别，重点关注：
● 是否输出结构化信息（段落/行/坐标/阅读顺序）
● 表格、多栏、复杂版式处理能力
预研输出：
● 是否满足“可定位（坐标级）”要求
● 并发处理能力评估（batch / GPU）

2️⃣ 文本定位与对齐（关键基础能力）
研究如何建立：
● OCR文本 ↔ 原文位置（坐标 / offset / 行ID）的映射关系
核心要求：
● 所有AI输出必须能回溯到原文
● 支持前端高亮、审核定位
预研输出：
● 定位数据结构设计（chunk / span / bbox）
● 是否支持页级 / 段落级 / 行级定位

3️⃣ 元数据抽取（规则 + RAG + LLM）
研究三种能力的组合方式：
● 规则匹配（regex / 模板）
● RAG（知识库辅助）
● LLM抽取（结构化输出）
重点不是“能不能抽”，而是：
● 如何提升准确率与稳定性
● 如何处理不确定性（置信度 / fallback）
预研输出：
● 抽取策略分层设计（规则优先 or LLM优先）
● 精度评估方案（人工校验机制）

4️⃣ 编目规则引擎（Rule Engine）
研究如何将编目逻辑从代码中剥离：
● 支持规则配置（JSON/YAML）
● 支持多分类体系、多策略
重点关注：
● 规则与AI结果如何融合（rule override / 校验）
核心价值： 支撑“多单位、多标准”的扩展能力
预研输出：
● 规则表达方式（DSL草案）
● 规则执行顺序设计（优先级机制）

5️⃣ 分布式处理 Pipeline（Ray方向）
研究是否采用 Ray 构建数据处理流水线：
● OCR → 解析 → 抽取 → 编目 → 质检
重点关注：
● DAG编排能力
● 任务重试 / 幂等
● 资源调度（CPU/GPU）
核心价值：决定系统是否能处理“海量文档”
预研输出：
● Ray vs Celery 对比结论
● Pipeline 拆分粒度设计（stage划分）


6️⃣ 算子（Operator）标准化
定义统一处理单元：
● 输入输出 schema 标准化
● 支持 DAG 组合
核心要求：
● 可复用（OCR / 抽取 / 分类）
● 可观测（日志 / metrics）
● 可重试（幂等）

核心价值： 避免系统演变成“不可维护的脚本集合”
预研输出：
● Operator接口定义（input/output）
● 示例算子（OCR算子 / 抽取算子）

7️⃣ LLM + Multi-Agent 审核系统
将复杂任务拆成多个 Agent：
● 抽取Agent / 分类Agent / 审核Agent / 仲裁Agent
重点研究：
● Agent协作模式（串行 / 投票 / 裁决）
● 输出必须包含：
    ○ evidence（依据）
    ○ reason（解释）
核心价值：提升系统“可解释性 + 可控性”
预研输出：
● Agent划分方案
● 一条完整审核链路设计

8️⃣ 敏感信息识别（规则 + 模型融合）
采用混合方案：
● regex（强规则）
● NER（实体识别）
● LLM（复杂语义）
重点：
● 定义统一敏感分类体系
● 控制误判率（precision优先）
预研输出：
● 敏感类型定义（分类体系）
● 多策略融合方案（优先级/投票）

9️⃣ 检索与RAG（服务编目与分析）
研究三类检索能力：
● 结构化查询（metadata）
● 全文检索（ES）
● 向量检索（RAG）
重点：
● 如何与“元数据抽取”打通
● 如何用于“编目纠错 / 辅助判断”
核心价值： 提升系统智能水平，而不是仅做存储
预研输出：
● chunk切分策略（页 / 段落）
● embedding方案选择

🔟 Data Agent（基于AI Agent的数据分析与统计）
要解决的问题
让用户可以通过自然语言直接完成：
● “今年某类档案数量是多少？”
● “不同部门归档数量对比”
● “异常档案（缺字段/低质量）有哪些？”
本质：
自然语言 → 查询计划 → 数据执行 → 结果解释
要看什么（重点预研）
1）NL2Query（自然语言转查询）
● Text2SQL（结构化数据）
● Query Planning（复杂查询拆解）
● Schema理解（表结构 + 字段语义）
2）Agent执行框架
● Tool Calling（调用SQL / ES / 向量检索）
● 多步推理（ReAct / Planner-Executor）
● 查询纠错与重试
3）数据语义层（关键）
● 指标定义（如“归档数量”）
● 统一口径（避免同一指标多种解释）
● Metadata + 语义映射
